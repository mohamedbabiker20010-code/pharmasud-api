### Overall Risk Assessment: **HIGH RISK**

PharmaSUD is a functional pharmacy POS system with several **critical security gaps** that must be addressed before production use with real customer/pharmacy data. The most severe issues involve authentication bypasses, missing authorization checks, wildcard CORS, and lack of rate limiting.

## 1. Authentication & Authorization

### [CRITICAL] 1.1 No Brute-Force Protection on Login
**File:** `auth.py` → `authenticate_user()`  
**Issue:** Login endpoint (`/api/auth/login`) has no rate limiting, account lockout, or CAPTCHA. Attackers can brute-force credentials unlimited times.
**Impact:** Credential stuffing, account takeover.
**Fix:** Add rate limiting (e.g., `slowapi` or middleware), max 5 attempts per IP/username per 15 min.

### [CRITICAL] 1.2 No Token Revocation / Logout Invalidation
**File:** `auth.py` → `create_access_token()`, `decode_token()`  
**Issue:** Tokens are stateless JWTs with 24-hour expiry. No blocklist/denylist for logout, password change, or admin disable. A stolen token works until expiry.
**Impact:** Stolen tokens remain valid; logout is client-side only (`localStorage.clear()`).
**Fix:** Implement Redis-based token denylist; check on each request; reduce token expiry to 1-2h + refresh tokens.

### [HIGH] 1.3 Password Policy Too Weak
**File:** `auth.py` → `create_admin_user()` (line 207), `AdminCreate` model (line 39)
**Issue:** Minimum 6 characters, no complexity requirements (no uppercase, number, symbol check).
**Impact:** Weak passwords easily guessed/cracked.
**Fix:** Enforce min 12 chars, mixed case, number, symbol; use `zxcvbn` for strength estimation.

### [HIGH] 1.4 No MFA / 2FA Support
**Impact:** Single-factor authentication only. High-value pharmacy data protected by password alone.
**Fix:** Add TOTP (Google Authenticator) or SMS/email OTP for admin accounts.

### [MEDIUM] 1.5 Session Management via localStorage Only
**File:** All templates (e.g., `shared_layout.html` line 263-264)
**Issue:** JWT stored in `localStorage` — vulnerable to XSS theft. No `HttpOnly` cookie option.
**Fix:** Store token in `HttpOnly; Secure; SameSite=Strict` cookie; keep minimal data in localStorage.

## 2. JWT / Token Handling

### [CRITICAL] 2.1 SECRET_KEY Generation on Render Only
**File:** `render.yaml` line 12-13
```yaml
- key: SECRET_KEY
  generateValue: true
```
**Issue:** `SECRET_KEY` is auto-generated by Render on first deploy. If service is recreated, **all existing tokens are invalidated** — users logged out. Worse: if `.env` is not set locally, `auth.py` line 30-31 raises `ValueError` at import time.
**Fix:** Use a fixed, strong key from env var; never auto-generate in production.

### [HIGH] 2.2 Algorithm Confusion Risk
**File:** `auth.py` line 32, 68
```python
ALGORITHM = "HS256"
payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
```
**Issue:** `algorithms=[ALGORITHM]` is correct (list), but no explicit `options={"verify_signature": True}`. Low risk currently but good practice.
**Fix:** Add `options={"verify_signature": True, "verify_exp": True, "verify_aud": False}`.

### [HIGH] 2.3 Token Payload Contains Excessive PII
**File:** `auth.py` line 317-322
```python
token_data = {
    "user_id": str(user.id),
    "pharmacy_id": str(user.pharmacy_id),
    "role": user.role,
    "username": user.username
}
```
**Issue:** Username in token. If token leaked, exposes identity. Acceptable but consider minimal claims.

### [MEDIUM] 2.4 24-Hour Token Expiry Too Long
**File:** `auth.py` line 33
```python
ACCESS_TOKEN_EXPIRE_HOURS = 24
```
**Impact:** Stolen token usable for 24h.
**Fix:** Reduce to 1-2 hours; implement refresh token rotation.

### [LOW] 2.5 `datetime.utcnow()` Deprecated
**File:** `auth.py` lines 56, 58
**Fix:** Use `datetime.now(timezone.utc)` (Python 3.11+).

if current_user.get("role") != "admin":

### [LOW] 3.4 Employee Can See All Users Including Admins

**Issue:** `list_employees` returns all users. No self-filtering or role-based visibility.

## 4. API Endpoint Security

### [CRITICAL] 4.1 Wildcard CORS — `allow_origins=["*"]`
**File:** `main.py` line 137-143
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```
**Impact:** **Any website** can make authenticated requests to your API (since `allow_credentials=True`). CSRF via credentialed requests possible. Browser will send cookies/auth headers to your API from malicious sites.
**Fix:** Restrict to your exact domain: `allow_origins=["https://pharmasud-api.onrender.com", "https://your-custom-domain.com"]`.

### [HIGH] 4.2 No Rate Limiting on Any Endpoint
**Impact:** API abuse, DoS, credential stuffing, scraping.
**Fix:** Add `slowapi` middleware; default 100 req/min per IP; stricter on auth (5/min).

### [HIGH] 4.3 Debug/Test Endpoints Exposed in Production
**File:** `main.py` line 358-454
```python
@app.get("/api/test/generate-sales-data")
def generate_test_sales(current_user: UserResponse = Depends(require_admin), ...)
```
**Issue:** Admin-only but creates 30 days of fake sales data. Dangerous if triggered accidentally. Also `/ping`, `/health` expose stack info.
**Fix:** Guard with `ENVIRONMENT != "production"` or separate feature flag; remove `/api/test/*` from prod.

### [HIGH] 4.4 `/scanner-debug` Page Requires Auth But No Admin Check
**File:** `main.py` line 255-258
```python
@app.get("/scanner-debug", response_class=HTMLResponse)
async def scanner_debug_page(request: Request, current_user: dict = Depends(get_current_user)):
```
**Issue:** Any logged-in user can access barcode scanner diagnostic page. Should be admin-only or removed in prod.

### [MEDIUM] 4.5 No API Versioning
**Impact:** Breaking changes will break frontend. All endpoints at `/api/*` root.

### [MEDIUM] 4.6 Missing Security Headers
**Issue:** No `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`.
**Fix:** Add `fastapi.middleware.trustedhost.TrustedHostMiddleware` and custom middleware for headers.

### [LOW] 4.7 `/api/auth/me` Returns `pharmacy_name`, `owner_name`
**File:** `main.py` line 336-347
**Issue:** Slightly excessive data exposure. Low risk.

## 5. SQL / Database Safety

### [HIGH] 5.1 Raw SQL with f-String Interpolation (SQL Injection Risk)
**Files:** Multiple — `reports.py` lines 86-87, 362, 395; `inventory.py` lines 86-87; `audit.py` lines 86-87
```python
# Example: audit.py line 86-87
rows = db.execute(
    text(f"""
        SELECT a.id, a.user_name, a.action_type, a.description,
               a.old_value, a.new_value, a.created_at
        FROM audit_log a
        {conditions}
        ORDER BY a.created_at DESC
        LIMIT :lim OFFSET :off
    """),
    {**params, "lim": limit, "off": offset}
).fetchall()
```
**Issue:** `conditions` built via string concatenation (`conditions += " AND a.action_type = :atype"`). While parameters are bound safely, the **query structure itself is dynamic** — if `action_type` came from user input without validation, could inject SQL. Currently `action_type` is from enum-like query param but not validated against allowlist.
**Fix:** Use allowlist validation for all dynamic query parts; prefer SQLAlchemy ORM/query builder.

### [HIGH] 5.2 Dynamic Table/Column Names in f-Strings
**File:** `reports.py` line 499-510 (monthly comparison query)
```python
monthly = db.execute(text("""
    SELECT DATE_TRUNC('month', s.created_at) as month,
           COALESCE(SUM(s.total_amount), 0) as revenue
    FROM sales s
    WHERE s.pharmacy_id = :pid
      AND s.created_at >= :start_date
    GROUP BY DATE_TRUNC('month', s.created_at)
    ORDER BY month DESC
    LIMIT 6
"""), params).fetchall()
```
**Actually OK** — static query. But audit.py and others use dynamic WHERE clauses.

### [MEDIUM] 5.3 No Prepared Statement Caching / Query Plan Optimization
**Impact:** Repeated raw SQL parsing overhead. Minor.

### [MEDIUM] 5.4 `DATABASE_URL` Only Checked at Import, Not Runtime
**File:** `database.py` line 21-23
```python
if not DATABASE_URL:
    raise RuntimeError("...")
```
**Issue:** If DB becomes unavailable at runtime, app crashes on requests. No circuit breaker or retry logic.
**Fix:** Add connection health checks; graceful degradation.

### [LOW] 5.5 `echo=False` — No SQL Logging in Production
**File:** `database.py` line 35
**Issue:** Hard to debug production issues. Enable structured logging for slow queries.

## 6. Input Validation & Sanitization

### [HIGH] 6.1 No Input Sanitization for Text Fields (XSS Risk in Rendered Content)
**Files:** All templates using `x-text` or `{{ }}` 
**Issue:** User-controlled data (medicine names, customer names, supplier names, batch numbers) rendered in HTML via Alpine.js `x-text` (safe) or Jinja2 `{{ }}` (auto-escaped). **BUT**: `x-text` sets `textContent` (safe), but some templates use `x-html` or raw interpolation?
**Check:** `shared_layout.html` uses `x-text` — safe. `medicines_list.html` uses `x-text`. **No `x-html` found** — good.
**BUT:** Audit log stores `description`, `old_value`, `new_value` from user input — if ever rendered without escaping, XSS.

### [HIGH] 6.2 File Upload — Image Validation Can Be Bypassed
**File:** `medicines.py` line 78-83
```python
ext = os.path.splitext(image_file.filename)[1].lower()
if ext not in ALLOWED_EXTENSIONS:
    raise HTTPException(...)
```
**Issue:** Only checks **file extension**, not **magic bytes**. Attacker uploads `shell.php.jpg` with PHP content — PIL may still process but could be served incorrectly.
**Fix:** Validate MIME type via `python-magic` or `filetype`; verify PIL can open; strip EXIF; rename to UUID.

### [MEDIUM] 6.3 Barcode Input Not Validated (Format/Length)
**File:** `models.py` line 128, `medicines.py` line 307
```python
barcode: Optional[str] = Field(None, max_length=50)
```
**Issue:** Accepts any 50-char string. No checksum validation (EAN-13, UPC-A). Could store garbage.

### [MEDIUM] 6.4 No Request Size Limits
**File:** `main.py` — no middleware for max body size.
**Impact:** Large payload DoS (e.g., huge JSON in sales create).
**Fix:** Add `starlette.middleware.gzip.GZipMiddleware` and body size limit middleware.

### [LOW] 6.5 Pydantic Validators Only on Some Models
**Consistency:** `MedicineCreate` has category validator; `BatchReceive` has no expiry date format validator (handled in endpoint).

## 7. File Upload Security

### [HIGH] 7.1 Images Stored as Base64 in Database
**File:** `medicines.py` line 73-111, `models.py` line 574
```python
image_path = Column(Text)  # Base64 image data
```
**Issue:** 
- Bloats DB (images in PostgreSQL = poor performance, backup size)
- No CDN/caching
- Base64 increases size ~33%
- If DB compromised, all images exposed
**Fix:** Store in object storage (S3, Cloudflare R2, Render Disk) — save only URL in DB.

### [MEDIUM] 7.2 No Virus/Malware Scanning on Uploads
**Impact:** User uploads malicious file (polyglot, exploit).
**Fix:** Integrate ClamAV or cloud scanning (e.g., AWS S3 + Lambda scanner).

### [LOW] 7.3 No Image Dimension/Size Limits Beyond 2MB
**File:** `medicines.py` line 38-40
```python
MAX_IMAGE_SIZE = 2 * 1024 * 1024
DEFAULT_IMAGE_SIZE = (300, 300)
```
**Acceptable** but consider max dimensions (e.g., 4000px) to prevent decompression bombs.

## 9. Frontend Security Issues

### [HIGH] 9.1 Alpine.js CSP Violation Risk
**File:** All templates load Alpine via CDN:
```html
<script defer src="https://cdn.jsdelivr.net/npm/alpinejs@3.x.x/dist/cdn.min.js"></script>
```
**Issue:** No **Subresource Integrity (SRI)** hash. If CDN compromised, malicious JS runs on all pages.
**Fix:** Add `integrity="sha384-..."` and `crossorigin="anonymous"`.

### [HIGH] 9.2 Lucide Icons Loaded Without SRI
**File:** All templates:
```html
<script src="https://unpkg.com/lucide@latest"></script>
```
**Issue:** `@latest` = **unpinned version**; no SRI. Supply chain attack risk.
**Fix:** Pin exact version; add SRI; self-host or use npm + build.

### [HIGH] 9.3 Google Fonts Loaded Without `crossorigin`
**File:** `shared_layout.html` line 8-10, `login.html` line 8-10
```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=IBM+Plex+Sans..." rel="stylesheet">
```
**Issue:** `preconnect` to `fonts.googleapis.com` without `crossorigin` leaks referrer; fonts loaded without SRI.

### [MEDIUM] 9.4 No Content Security Policy (CSP)
**Impact:** No defense-in-depth against XSS, mixed content, unauthorized scripts.
**Fix:** Implement CSP header (start with `report-only`):
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net https://unpkg.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; img-src 'self' data:; connect-src 'self' https://pharmasud-api.onrender.com;
```

### [MEDIUM] 9.5 `localStorage` Used for Auth Token (XSS Theft)
**Covered in 1.5.**

### [MEDIUM] 9.6 No `X-Frame-Options` / `frame-ancestors` — Clickjacking Risk
**Fix:** Add `X-Frame-Options: DENY` or CSP `frame-ancestors 'none'`.

### [LOW] 9.7 Inline Styles & Scripts (CSP `unsafe-inline` Required)
**File:** All templates have large `<style>` and `<script>` blocks.
**Fix:** Move to static files; use nonces/hashes in CSP.

### [LOW] 9.8 Dark Mode Preference in localStorage — Low Risk
**Ok.**

## 10. CSRF, XSS, Injection Risks

### [HIGH] 10.1 No CSRF Protection
**Issue:** State-changing endpoints (POST/PUT/DELETE) accept JSON with `Authorization: Bearer` header — **no CSRF token**. Since tokens in `localStorage` (not cookies), **CSRF not directly applicable** for API calls. **BUT**: If cookies used later, or if `allow_credentials=True` with wildcard CORS (4.1), **CSRF via credentialed requests is possible**.
**Fix:** Fix CORS first; then add CSRF tokens for any cookie-based auth.

### [HIGH] 10.2 Reflected XSS via Error Messages
**File:** `main.py` line 278, 480 — returns `str(e)` in error responses
```python
message=f"حدث خطأ أثناء معالجة البيع: {str(e)}"
```
**Issue:** If exception contains user input, reflected in JSON. Frontend displays via `x-text` (safe) but if ever `x-html` used, XSS.
**Fix:** Sanitize error messages; never return raw exception text to client.

### [MEDIUM] 10.3 No Input Validation on Search Queries (ReDoS Potential)
**File:** `medicines.py` line 272-277 — `ilike` with user input
```python
search_term = f"%{search}%"
query = query.filter((Medicine.trade_name.ilike(search_term)) | ...)
```
**Issue:** SQLAlchemy parameterizes — safe from SQLi. But wildcard prefix `%` on unindexed column = slow query (DoS via complex search).
**Fix:** Add trigram index (`pg_trgm`); limit search length.

### [LOW] 10.4 Batch Number Uniqueness Check Race Condition
**File:** `batches.py` line 157-162
```python
existing = db.query(Batch).filter(Batch.batch_number == data.batch_number, ...).first()
if existing: raise HTTPException(...)
# then INSERT
```
**Issue:** TOCTOU — two concurrent requests can both pass check, then both insert duplicate. DB has no unique constraint on `batch_number` (only per medicine? No, global check in code).
**Fix:** Add `UNIQUE(batch_number)` constraint in DB; catch `IntegrityError`.

## 14. Production Deployment Readiness

### [CRITICAL] 14.1 No WAF / DDoS Protection
**Render** provides basic DDoS but no WAF rules (SQLi, XSS, bot detection).
**Fix:** Enable Cloudflare (free tier) in front of Render; configure WAF rules.

### [HIGH] 14.2 Single-Region Deployment (Render Free Tier)
**Issue:** No HA, no failover. DB on same region. If region down, service down.
**Fix:** Document RPO/RTO; consider paid plan for multi-AZ or backup region.

### [HIGH] 14.3 No Automated Backups Verification
**Render Postgres** does daily backups (free tier: 7 days). **But**: No restore testing documented.
**Fix:** Quarterly restore drill; document procedure.

### [MEDIUM] 14.4 No CI/CD Pipeline (GitHub Actions)
**Issue:** Direct push to `master` → auto-deploy on Render. No tests, lint, security scan.
**Fix:** Add GitHub Actions: `ruff`, `mypy`, `pytest`, `pip-audit`, `bandit`, deploy on green.

### [MEDIUM] 14.5 No Database Migration System
**File:** `main.py` lines 45-135 — **raw DDL in startup event**
```python
@app.on_event("startup")
async def create_tables():
    Base.metadata.create_all(bind=engine)
    # ALTER TABLE ... ADD COLUMN IF NOT EXISTS ...
```
**Issue:** 
- `create_all` only creates missing tables — **no schema evolution**
- Raw `ALTER TABLE` for migrations — not idempotent, no rollback, no versioning
- Concurrent deploys can race
**Fix:** Use **Alembic** for migrations; run as separate step before deploy.

### [LOW] 14.6 No Staging Environment
**Fix:** Create `staging` Render service from `staging` branch.
